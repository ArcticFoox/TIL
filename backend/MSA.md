# MSA Architecture

## 모놀리식 아키텍처(Monolithic Architecture, MA)

하나의 통합된 코드 베이스로 여러 비즈니스 기능을 수행하는 전통적인 소프트웨어 개발 모델로, 단일 애플리케이션 내에 서비스의 모든 로직이 통으로 들어가 있는 구조

가장 큰 장점은 간결하다는 점으로, 중앙 집중된 구조이기 때문에 분산된 애플리케이션에 비해 엔드 투 엔드 테스트(End-to-End, E2E: 사용자 관점에서 앱의 처음부터 끝까지 테스트)를 더 빠르게 수행할 수 있음

단일 앱에 비즈니스 로직부터 UI, 콘텐츠 등 모든 구성 요소를 이루는 코드가 들어있기 떄문에 디버깅도 간편

단점으로 단일 앱이 커지면 자연히 구동부터 빌드, 배포에 드는 시간이 오래 걸리고, 하나로 된 거대한 시스템 구조를 제대로 파악하지 않으면 특정 컴포넌트나 모듈에서 발생하는 성능 문제나 장애가 다른 영역에까지 영향을 주게 됨

아무리 작은 부분만 수정하더라도 전체 애플리케이션을 통째로 컴파일해서 배포해야 하는 만큼 배포가 잦은 환경에서는 번거로움 발생하고 기능별로 알맞은 기술이나 언어, 프레임워크를 따로 선택하기 까다로움

## 마이크로서비스 아키텍처(MicroService Architecture, MSA)

아주 작은 서비스 단위로 나눠 각 서비스에서 독립적으로 서비스를 구성하는 모델

중앙 집중적인 관리 체계 대신 경량화된 API나 메시지로 직접 통신하면 접근하는 방식을 취함

장점은 서비스를 잘게 쪼개면서 개발 구조가 민첨하고 유연해졌다는 점으로 각 서비스를 독립적으로 개발하고 배포할 수 있어서 작업 시간이 단축되고 확장성도 높음

한 서비스가 다운되더라도 다른 서비스는 문제없이 작동할 수 있음

서비스에 적합한 언어나 프레임워크 등 기술을 독립적으로 선택할 수 있고, 자체 DB를 서비스마다 가지고 있어서 데이터 무결성을 유지하는 데도 도움이 됨

단점으로 분산된 서비스가 서로 API를 호출하는 과정에서 통신 비용과 지연 시간이 들고, 큰 인프라 비용이 발생할 수 있음

장애 추적이나 디버깅, E2E 테스트 역시 쉽지 않음

DB가 분리되면서 데이터를 조회하기 어렵고 DB간 데이터 중복이 발생할 수 있음

서비스가 커지는 만큼 복잡해지기 때문에 적절하게 관리하지 않으면 오히려 개발 속도나 운영 성능이 나빠질 수 있음
